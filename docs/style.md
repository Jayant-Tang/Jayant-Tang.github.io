# 博客写作风格指南

本文档基于当前仓库 `source/_posts` 下的全部历史文章提炼，用于后续生成新博客时保持一致的表达、结构和技术深度。

## 1. 核心定位

本博客是面向嵌入式、Nordic、Zephyr/NCS、Matter、Wi-Fi、Linux 工具链等方向的工程实践型技术博客。

写作目标：帮助读者真正完成一件事。概念介绍只作为铺垫：

1. 理解背景和原理；
2. 配好环境或代码；
3. 编译、烧录、运行、抓日志；
4. 能在遇到问题时知道该查哪里、怎么定位。

文章应体现以下特点：

- **工程实战优先**：尽量落到具体 SDK 版本、开发板、配置文件、命令、日志、截图、测试结果。
- **先搭桥再深入**：先用直观语言解释概念，再逐步进入 Kconfig、Devicetree、CMake、驱动源码、协议细节。
- **把抽象概念落到代码**：遇到框架、协议、配置系统时，要说明它最终如何影响编译、固件、设备行为或寄存器。
- **保留排障线索**：解释为什么这样配置、错了会怎样、如何验证是否生效。
- **不写空泛教程**：避免只罗列官方文档步骤，要加入自己的理解、国内环境踩坑、实测结果、注意事项。

## 2. 读者假设

默认读者是有一定嵌入式或软件开发基础，但不一定熟悉当前主题的人。

可以假设读者知道：

- C/C++ 基础；
- 命令行基本操作；
- Git、VS Code、串口、编译、烧录等基本概念；
- 单片机、RTOS、网络协议的基础术语。

不能假设读者已经理解：

- Zephyr/NCS 的 Kconfig、Devicetree、Sysbuild、Partition Manager；
- Nordic 特有外设，如 PPI/DPPI、GPIOTE、UARTE、UICR、Modem；
- Matter 的证书体系、Data Model、CHIP Tool、OTA 流程；
- Wi-Fi 抓包、TLS 证书、MQTT、nRF Cloud 等具体工作流。

遇到这些内容时，要用少量篇幅补齐概念，再进入实践。

## 3. 文章整体结构

### 3.1 常用开头

文章开头通常不寒暄，直接说明本文要解决的问题。

常见形式：

```markdown
# 1. 简介

本文介绍如何...
本文将以...为例，展示...
本文基于 NCS vX.Y.Z，介绍...
```

或者：

```markdown
# 1. 前言

之前写过...，但很多读者在实际使用时还会遇到...
本文就通过...这样一个例子，来实际感受...
```

开头应包含：

- 本文主题；
- 使用的平台、SDK 版本、开发板或操作系统；
- 本文会覆盖哪些内容；
- 如果有限制，提前说明。

示例：

```markdown
本文基于 nRF Connect SDK v3.0.2，介绍如何在 Nordic Matter 工程中添加自己的蓝牙广播和 GATT 服务。

本文包含以下内容：

- 多蓝牙广播方案分析
- 蓝牙身份与地址配置
- Matter 广播仲裁器的使用
- 代码修改和运行验证
```

### 3.2 推荐大纲

多数文章采用编号标题，一级标题从 `# 1.` 开始。

常见结构：

```markdown
# 1. 简介 / 前言 / 本文主题
# 2. 背景概念 / 原理说明
# 3. 环境准备 / 硬件准备 / 软件准备
# 4. 配置、编译与烧录
# 5. 运行验证 / 效果演示
# 6. 代码解析 / 原理解析
# 7. 常见问题 / 踩坑记录 / 注意事项
# 8. 总结 / 参考链接
```

对于大型深度文章，可以按“概念 → 配置 → 源码 → 实测 → 应用开发”展开。

对于短教程，可以直接：

```markdown
# 1. 概览
# 2. 配置
# 3. 验证
# 4. 注意事项
```

### 3.3 标题风格

标题要直接描述内容，不追求文学化。

推荐：

- `# 1. 简介`
- `# 2. 环境准备`
- `## 设备树 overlay`
- `## 运行并测试`
- `## 代码解析`
- `## 常见问题`

可接受：

- `## 为什么需要 Nordic MCP？`
- `## 什么是 MQTT 协议？`
- `## Zephyr 设备电源管理（PM_DEVICE）`

避免：

- “一篇文章彻底搞懂...”
- “保姆级教程...”
- “震惊！原来...”
- 过度营销和标题党。

## 4. 内容展开方式

### 4.1 先给结论，再讲细节

遇到配置项、工具、方案选择时，先告诉读者应该怎么做，再解释为什么。

推荐写法：

````markdown
推荐直接开启以下配置：

```shell
CONFIG_PM_DEVICE_RUNTIME=y
CONFIG_PM_DEVICE=y
```

这两个配置分别对应 Zephyr 的 Device Power Management 和 Runtime Device Power Management。前者提供手动挂起/恢复外设的能力，后者让驱动在每次使用外设前后自动管理引用计数。
````

### 4.2 解释要落到工程动作

解释概念时，不停留在定义，要说明它会影响什么文件、什么代码、什么运行结果。

例如解释 Kconfig：

- 不只说“Kconfig 是配置系统”；
- 要说它定义了菜单、配置项会合并到 `.config`、会生成 `autoconf.h`、CMake 会根据配置项决定源码是否参与编译。

例如解释 Devicetree：

- 不只说“描述硬件”；
- 要说最终会生成 `zephyr.dts` 和 `devicetree_generated.h`，驱动通过 compatible/status 找节点并创建 device 实例。

### 4.3 用“问题意识”组织原理

原理部分要从读者会疑惑的问题切入。

常用句式：

- “很多新上手 Zephyr 的读者会有疑惑：...”
- “这里会产生一个问题：...”
- “为什么只改了 config，设备树就不用改了？”
- “我们自然可以联想到...”
- “这就解释了为什么...”

### 4.4 保留个人实测和判断

文章可以出现“我这里”“我实测”“我推荐”“我不推荐”。

这类表达是博客风格的一部分，但要有依据：

```markdown
在 nRF54L15 上，不推荐继续使用 Timer + PPI 的方式做硬件计数。经过我实测，开启 FRAMETIMEOUT 时会出现超时不生效的问题。
```

避免无依据的主观判断。

## 5. 语言风格

### 5.1 中文表达

语言应自然、直接、工程化。技术术语保留英文。

推荐：

- “这个配置项的作用是...”
- “这样做的好处是...”
- “这里有两个注意点...”
- “如果你只是想快速验证，可以...”
- “实际产品中不建议这样做...”

避免：

- 翻译腔；
- “作为一个...来说”；
- “综上所述我们可以发现”这类空泛连接；
- 过度客套和 AI 味；
- “这不是 X，而是 Y”“不是为了 X，而是为了 Y”“不仅是 X，更是 Y”这类模板化对比句。直接写结论，不要先立一个靶子再转折。

更自然的改法：

- 把“这不是一个配置问题，而是驱动初始化顺序问题”改成“根因在驱动初始化顺序”；
- 把“本文不是介绍概念，而是带你实战”改成“本文按实战流程展开，概念只在必要处补充”；
- 把“不仅要修改 Kconfig，更要修改 Devicetree”改成“Kconfig 和 Devicetree 都要改”。

### 5.2 术语处理

英文技术术语保留英文，并在第一次出现时解释。

例如：

- `Kconfig`、`Devicetree`、`DeviceTree`、`CMake`、`Sysbuild`
- `Matter Controller`、`Thread Border Router`
- `MQTT Broker`、`Topic`、`TLS`
- `GPIOTE`、`PPI/DPPI`、`UARTE`、`UICR`
- `Factory Data`、`DAC`、`PAI`、`PAA`

首次出现时可写：

```markdown
Thread 边界路由器（Border Router）的作用是衔接 Thread 网络和外部网络。
```

### 5.3 常用表达习惯

可使用以下表达：

- “简单来说，...”
- “从软件开发角度看，...”
- “从硬件角度看，...”
- “这里先不展开，后文会解释。”
- “这部分了解即可，实际开发中...”
- “不要直接修改 SDK 里的文件，应该在自己的工程中用 overlay 覆盖。”
- “如果只是测试，可以这样；实际产品中应当...”

少用但可保留的表达：

- “多是一件美事”
- “照葫芦画瓢”
- “踩坑”
- “坑点”

但不要刻意模仿口头禅，重点是清楚。

## 6. 代码、命令和配置规范

### 6.1 代码块要有明确上下文

给代码前先说明文件路径和用途。

推荐：

````markdown
配置文件：`prj.conf`

```shell
CONFIG_LOG=y
CONFIG_LOG_BACKEND_UART=y
```
````

````markdown
设备树 overlay：

```dts
&uart0 {
    status = "okay";
};
```
````

### 6.2 配置项必须解释作用

不要只贴配置。贴完后用列表解释关键项。

```markdown
说明：

- `CONFIG_SERIAL=y`：使能 Zephyr 标准串口驱动；
- `CONFIG_UART_ASYNC_API=y`：开启异步串口 API；
- `CONFIG_UART_0_ASYNC=y`：让 Nordic 的 uart0 实例使用异步 API。
```

### 6.3 命令要标明环境

命令行代码块尽量标注 shell 类型。

- Windows 使用 `powershell`
- Linux/macOS 使用 `bash`
- 通用命令可用 `bash`
- 配置文件用 `shell`、`conf`、`yaml`、`cmake`、`dts` 等

示例：

```powershell
nrfutil toolchain-manager launch --ncs-version v3.2.4 -- powershell
```

```bash
west build -d build -b nrf52840dk/nrf52840 --sysbuild
```

### 6.4 注释风格

代码注释可以中英文混用，但中文读者导向文章中优先使用中文注释。

注释要解释意图，不要只翻译代码。

推荐：

```c
// 关闭 RX，避免串口一直处于接收状态导致功耗升高
uart_rx_disable(uart_dev);
```

不推荐：

```c
// 调用函数
uart_rx_disable(uart_dev);
```

### 6.5 截图与日志

文章中大量使用截图是正常风格。截图通常用于：

- 展示 VS Code/nRF Connect 图形界面；
- 展示编译结果；
- 展示串口日志；
- 展示功耗波形；
- 展示 Wireshark、MQTTX、CHIP Tool 等工具界面；
- 展示芯片手册或官方文档截图。

截图后要写一句解释，不要只贴图。

```markdown
可以看到设备成功连接 Wi-Fi，并开始连接 MQTT Broker：

![...](...)
```

### 6.6 表格使用

适合用表格的场景：

- 软件/硬件清单；
- 参数说明；
- 协议字段；
- 不同方案对比；
- 配置项来源；
- 工厂数据字段；
- 功耗或资源占用结果。

表格内容要尽量简短，复杂解释放到表格后。

## 7. 注意、备注和折叠块

### 7.1 引用块用于强调风险

常用格式：

```markdown
> 注意：
>
> - ...
> - ...
```

或：

```markdown
> 【注意】
> 本文介绍的方法仅支持 NCS v2.8.0 之后的版本。
```

适合放在引用块中的内容：

- 版本限制；
- 安全风险；
- 硬件损坏风险；
- 网络环境限制；
- 不推荐做法；
- 踩坑记录；
- 国内网络特殊处理。

### 7.2 更新记录

对于长期维护型文章，Front Matter 后可以放更新记录：

```markdown
> 2026.1.11 更新：
>
> - 官方已经支持中国大陆服务器源下载 NCS
>
> 2025.7.27 更新：
>
> - 增加 nrfutil 详细安装说明
```

更新记录适合放在文章最开头，不影响正文编号。

### 7.3 折叠块

较长但非主线的内容可以用 `<details>`。

适合折叠：

- 老版本兼容方案；
- 大段报错日志；
- 离线安装方法；
- 已弃用方案；
- 额外背景知识。

示例：

```html
<details>
    <summary>[点击展开]</summary>

这里放较长的补充内容。

</details>
```

## 8. Hexo 与 Front Matter 规范

新文章应使用 Hexo Front Matter。

推荐模板：

```yaml
---
title: 文章标题
typora-root-url: ./..
typora-copy-images-to: ../../source/imgs/${filename}.assets/
date: YYYY-MM-DD HH:mm:ss
cover:
tags:
  - Nordic
  - Zephyr
categories: Zephyr
---
```

规则：

1. `title` 使用中文技术标题，必要时保留英文术语；
2. `tags` 使用技术关键词，如 `Nordic`、`Zephyr`、`NCS`、`Matter`、`WiFi`；
3. `categories` 使用大类，如 `Nordic`、`Zephyr`、`Matter`、`WiFi`、`Linux`、`AI`；
4. `typora-root-url` 和 `typora-copy-images-to` 保持现有仓库习惯；
5. 未完成文章可加 `published: false`；
6. 归档文章可加 `archive: true`；
7. 重要文章可加 `sticky`，但不要滥用。

## 9. 不同类型文章模板

### 9.1 原理详解型

适合主题：Devicetree、Kconfig、BLE 地址、Matter 证书、GPIO 硬件原理。

结构：

```markdown
# 1. 前言
# 2. 基本概念
# 3. 核心机制
# 4. 在 NCS/Zephyr 中的实现
# 5. 代码/API/配置方法
# 6. 实际应用与注意事项
# 7. 总结
```

写作重点：

- 从简单模型解释；
- 用图、表、代码说明；
- 最后落到实际 API、配置文件、源码路径。

### 9.2 实战教程型

适合主题：安装环境、运行例程、烧录固件、抓包、配置开发工具。

结构：

```markdown
# 1. 简介 / 概览
# 2. 硬件准备
# 3. 软件准备
# 4. 配置、编译并烧录
# 5. 运行并测试
# 6. 常见问题
# 7. 参考链接
```

写作重点：

- 每一步都要可执行；
- 给出具体命令和截图；
- 明确 Windows/Linux 差异；
- 写清楚失败时如何检查。

### 9.3 例程代码解析型

适合主题：NCS Matter 例程、MQTT 例程、Asset Tracker、串口例程。

结构：

```markdown
# 1. 简介
# 2. 运行例程
# 3. 工程结构
# 4. 编译与配置系统
# 5. 关键模块解析
# 6. 数据流 / 状态机 / 线程模型
# 7. 如何改成自己的应用
# 8. 总结
```

写作重点：

- 先让例程跑起来；
- 再解释目录结构、Kconfig、CMake；
- 再解释模块间通信；
- 最后告诉读者哪些代码可以删、哪些必须保留、怎么改。

### 9.4 踩坑记录型

适合主题：功耗异常、Windows 编码问题、west update 报错、USB/串口/网络问题。

结构：

```markdown
# 1. 问题现象
# 2. 复现环境
# 3. 排查过程
# 4. 根因分析
# 5. 解决方法
# 6. 总结
```

写作重点：

- 必须写清楚版本、硬件、现象；
- 保留关键错误日志；
- 不要只给最终答案，要写排查路径；
- 明确哪些是推测，哪些是实测结论。

### 9.5 方案选型型

适合主题：Matter 多 BLE 广播方案、Add-on 安装方式、OTA 升级方式、网络拓扑。

结构：

```markdown
# 1. 背景
# 2. 方案一
# 3. 方案二
# 4. 方案对比
# 5. 推荐选择
# 6. 实现示例
```

写作重点：

- 先定义约束条件；
- 每种方案都要说明优缺点；
- 给出推荐场景；
- 如果方案会改 SDK 源码，要明确风险。

## 10. 技术准确性要求

生成新博客时必须遵守：

1. **版本明确**：涉及 SDK、协议、工具时标注版本，如 `NCS v3.0.2`、`Zephyr 4.x`、`Matter 1.3`。
2. **不确定就写不确定**：如果只是推测，使用“我猜想”“可能”“需要进一步验证”。
3. **配置要闭环**：给出 Kconfig 后，尽量说明它来自哪里、影响什么。
4. **源码路径要具体**：引用 SDK 源码时，给出类似 `${NCS}/zephyr/drivers/...` 的路径。
5. **运行结果要可验证**：尽量提供日志、截图、功耗数据、命令输出或现象描述。
6. **安全信息要提醒**：API Key、私钥、证书、工厂数据、设备唯一 ID 等必须提醒不要泄露。
7. **不要改 SDK 作为首选方案**：除非确实需要，否则优先用工程内 overlay、Kconfig、CMake、wrapper 方式实现。
8. **国内网络环境要单独说明**：涉及 GitHub、DockerHub、OAuth、官方服务器时，说明国内可行方案或网络要求。

## 11. 常见写法示例

### 11.1 “配置 + 解释”模式

````markdown
在 `prj.conf` 中添加：

```shell
CONFIG_BT_EXT_ADV=y
CONFIG_BT_EXT_ADV_MAX_ADV_SET=2
CONFIG_BT_MAX_CONN=2
```

其中：

- `CONFIG_BT_EXT_ADV=y`：开启扩展广播 API；
- `CONFIG_BT_EXT_ADV_MAX_ADV_SET=2`：允许同时创建两个广播集；
- `CONFIG_BT_MAX_CONN=2`：允许两个 BLE 连接同时存在。
````

### 11.2 “现象 + 根因”模式

```markdown
现象是，程序可以正常打印日志，但底电流一直维持在几百 uA。

这通常说明 CPU 已经进入 IDLE，但某个外设没有关闭。对于串口日志来说，优先检查两个地方：

1. 是否启用了异步串口；
2. 对应 UART 节点是否开启了 `zephyr,pm-device-runtime-auto;`。
```

### 11.3 “源码落地”模式

```markdown
关于这个问题，不要只看文档，最好直接看驱动代码。

在 `${NCS}/zephyr/drivers/serial/uart_nrfx_uarte.c` 中，可以看到驱动在发送前后调用了 Runtime PM 相关接口。因此，是否能低功耗，取决于驱动是否进入了这条路径。
```

## 12. 避免事项

写新文章时避免：

- 只翻译官方文档，没有自己的实验和解释；
- 只贴命令，不解释参数；
- 只解释原理，不给工程落点；
- 大量空泛形容词，如“非常强大”“十分方便”，但不说明哪里强、哪里方便；
- 不说明版本，导致读者照做失败；
- 混用不同 NCS 版本的 board target、路径、配置项；
- 把临时测试用的 API Key、证书、私钥、真实邮箱、真实客户信息写进文章；
- 使用“保姆级”“全网最细”“小白必看”等标题党表达；
- 为兼容所有场景写一大堆无用分支。博客应针对一个明确场景，把这条路径写清楚。

## 13. 生成新博客时的工作流

后续使用本风格生成文章时，建议按以下流程：

1. 先确认主题、目标读者、SDK/工具版本、硬件平台；
2. 决定文章类型：原理详解、实战教程、例程解析、踩坑记录或方案选型；
3. 先写大纲，再填充命令、配置、代码和验证结果；
4. 每个配置块后补充“为什么这样配”；
5. 每个操作流程后补充“如何确认成功”；
6. 最后写“注意事项”或“总结”，避免泛泛收尾。

如果信息不足，不要编造截图、日志或实测结果。可以写成：

```markdown
这一部分我还没有实测，后续补充。
```

或者：

```markdown
以下内容基于官方文档推断，实际项目中建议再用开发板验证。
```
