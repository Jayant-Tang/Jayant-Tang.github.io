# 1. 概述

本文讲解本博客当前使用的搭建方式，并同步更新到 2026 年我这套实际可用的版本和部署流程。

| 部件                           | 作用                                    |
| ------------------------------ | --------------------------------------- |
| [Hexo](https://hexo.io/zh-cn/) | 把.md文章转换为页面，并且快速部署到平台 |
| GitHub Pages                   | 提供从GitHub仓库创建网站的服务          |
| GitHub Actions                 | GitHub提供的免费持续集成服务            |

本文示例环境如下：

| 项目     | 当前版本 / 说明         |
| -------- | ----------------------- |
| Node.js  | 24.5.0                  |
| Hexo     | 8.1.2                   |
| hexo-cli | 4.3.2                   |
| 源码仓库 | `my-hexo`               |
| 发布仓库 | `Jayant-Tang.github.io` |

## Hexo + Github Pages

先从简单的讲起，看图：

<img src="https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined61b1017d6bdd08d92527797a4c89f235.png" alt="image-20221202224730111" style="zoom: 50%;" />

简而言之：

- Hexo是一个博客框架，存放在你的电脑上，能把你的markdown文章转换成网站源码，并存入一个git仓库
- Github Pages是GitHub提供的免费静态网站服务，能让你展示文档。你只需要把网站源码存入一个repository，并且启用这个服务，GitHub就会帮你建一个静态网站，分配一个网址。别人从这个网址就能看到你的博客了。
- Hexo能一键把本地的网站源码推送到GitHub，让GitHub自动将其部署为Pages

## GitHub Actions

​    前面的方法，网站所有的资源都存在本地，除了markdown文章和图片这种**资源文件**以外，网站的**美化主题**、**配置**等也存在本地。

​    这会产生几个问题：

- 数据容易丢失，一旦硬盘损坏，你的博客就灰飞烟灭了
- 只能在一台电脑上写文章，多台电脑之间hexo文件夹不好同步

- 网站的配置文件修改，没有历史记录追踪
- 由于网站的美化主题本身就是Github上的项目，每次要手动更新，很麻烦
- 每次写完文章，都要用hexo命令生成网站，还要用hexo命令部署网站，重复操作

​    所以，**最好把hexo文件夹本身也当作Git项目，并放在GitHub托管**。并且，还要把hexo部署的操作给自动化，让我们每次写完markdown文章，一 Push 就能自动部署页面。

​    这就要用到GitHub Actions，它是GitHub提供的**持续集成**（Continuous integration， CI）服务。这里不详述CI/CD的概念，你只需要知道，GitHub提供的这个服务，可以让你的仓库在特定条件下（例如push时），触发对应的**动作（Actions）**。

​    所谓的Actions，可以认为就是一条一条的Linux命令（也可以是Windows CMD命令）或者脚本，并且GitHub给你一个虚拟服务器去自动执行这一条一条的命令，这个虚拟服务器具有2核CPU，7GB内存，14GB硬盘。

​    你可以写自己的Actions，也可以用别人的Actions。Github提供了一个Action商店。

​    最终的效果应该如下图：

<img src="https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined305a64ad66a65ae0356fd8704191fb55.png" alt="image-20221203224217145.png" style="zoom: 50%;" />

配置完成后的效果：

- 写完文章后，直接本地hexo仓库`commit`，然后`push`
- GitHub上的hexo仓库由`push`触发Actions，自动在虚拟服务器中生成网站源码，然后部署到网站仓库中
- 网站仓库（xxxx.github.io）更新时，自动更新Github Pages
- hexo的美化主题，作为hexo的git submodule，每次push时自动checkout最新分支



# 2. 搭建步骤

一些简单的步骤就只简略提一下。

## 2.1. 在GitHub创建仓库并启用Pages

1. 注册一个 GitHub 账号
2. 先分清楚 GitHub Pages 的两种模式：

   - **个人页（user page）**：GitHub 仓库名必须是`<用户名>.github.io`，网址是`https://<用户名>.github.io/`
   - **项目页（project page）**：GitHub 仓库名可以随意，例如仓库名叫`blog`，网址就是`https://<用户名>.github.io/blog/`

   这两种模式最关键的区别，不是“能不能部署”，而是**最终网址根路径不同**。

3. 我的当前做法是创建两个仓库：

   - `Jayant-Tang.github.io`：真正用于发布博客，这是**个人页**
   - `my-hexo`：Hexo 源码仓库，里面放文章、主题配置、工作流等

   两个仓库都可以先留空，不创建任何文件。

4. 然后设置 Pages，在发布仓库中：

![image-20221202233043747](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined8f02bba63152a47b28dcd81c8a617607.png)

在仓库的 Settings 中，选择 Pages，下方选择哪个分支的哪个目录被识别为网站的根目录。

> GitHub 默认主分支通常是`main`，但你也可以继续使用`master`。关键不是分支名，而是 Pages 配置、工作流配置和 Hexo 的 deploy 配置要保持一致。
>
> 如果你之前已经使用过项目页，例如`https://<用户名>.github.io/<旧仓库名>/`，后面又迁移到了个人页，那么旧仓库可以只保留一个跳转页，让旧链接自动跳到新站。

## 2.2. 本地部署hexo

### 安装软件

首先安装依赖，需要安装的软件：

- [Git (git-scm.com)](https://git-scm.com/)：版本管理
- [Node.js](https://nodejs.org/en/download/)：hexo所需的平台

>Windows用户安装时，一定要勾选`Add to PATH `选项

然后安装hexo

``` bash
npm  install -g hexo-cli # 全局安装
```

>关于Node.js和hexo的版本依赖关系，请参考：[文档 | Hexo](https://hexo.io/zh-cn/docs/#Node-js-版本限制)



### 初始化hexo

找个喜欢的地方创建一个文件夹用于存放hexo项目，例如`my-hexo`。

进入这个仓库，然后进行hexo初始化

``` bash
# 新建文件夹
$ mkdir my-hexo && cd my-hexo

# 初始化hexo，执行这步之前，需确保文件夹是空的
$ hexo init ./

# 安装依赖的node.js插件
$ npm install
```

> 注意：
>
> - hexo初始化要求文件夹为空，因此必须先初始化hexo，再初始化git。否则`.git/`的存在将导致hexo无法初始化
> - `hexo init`时，产生了文件`package.json`，里面记录了hexo所依赖的npm包的具体来源和版本号
> - `npm install`时，就根据`package.json`记录的内容进行了包的安装。包的具体安装情况记录在`package-lock.json`中

安装完毕后，目录结构应该如下

```
.
|-- _config.landscape.yml  # landscape主题配置文件
|-- _config.yml            # 网站配置文件
|-- node_modules           # node.js插件包
|-- package-lock.json      # node.js插件包记录
|-- package.json           # node.js插件包记录
|-- scaffolds/             # 文章、页面模板文件夹
|-- source/                # 网站静态资源（文章、图片等）
`-- themes/                # 网站美化主题文件夹
```



### 初始化git并传到远程仓库

>如果你从没用过git，需要配置本地的git用户和邮箱：
>
>```bash
>$ git config --global user.name <用户名>
>$ git config --global user.email <邮箱>
>```
>
>这个用户名和邮箱只是本地提交代码时记录修改人是谁，并非是要登录什么网站

​    还是在这个目录，进行Git仓库初始化

```bash
git init
```

> 本文不解释git及其相关概念，后续使用git命令时，即使是复制，也务必明白你在做什么
>
> 另外，记得确认仓库中的`.gitignore`文件，它是`hexo init <文件夹>`时生成的，其内容包括：
>
> ```.gitignore
> .DS_Store
> Thumbs.db
> db.json
> *.log
> node_modules/         # 安装的npm包
> public/               # 生成的静态网站源码目录
> .deploy*/             # xxxx.github.io仓库
> _multiconfig.yml
> ```
>
> ​    这是让git不要去追踪一些自动生成的内容，这些内容与文章、站点配置无关。

​    然后再`commit`：

```bash
$ git add .
$ git commit -m "init"
```

​    还记得前面在GitHub创建的`my-hexo`仓库还是空的吗，现在需要把本地`my-hexo`仓库绑定到GitHub仓库。

```bash
$ git remote add origin https://github.com/<用户名>/<仓库名>.git
$ git branch -M master  # 默认主分支我改名为master
$ git push -u origin master
```

## 2.3.  配置hexo

### 安装主题

​    hexo有很多美化主题，可以在[Themes | Hexo](https://hexo.io/themes/)找到喜欢的主题，也可以在GitHub上搜索其他主题。**我当前博客实际使用的是 Fluid 主题**，不过这里仍以 Butterfly 的安装流程举例，因为主题安装思路本质上是相通的：安装主题、拷贝主题配置、在 Hexo 主配置中启用主题。

​    主题放在themes文件夹下。**不论安装什么主题，一定要注意**：

- 虽然主题的官方教程一般都是让用`git clone`到themes文件夹下，但是我们的hexo文件夹已经是一个git仓库了，所以要以`git submodule`的形式安装；
- `themes`下的主题文件夹必须以`<主题名>`来命名，而不是GitHub仓库的名称，如`hexo-theme-<主题名>`。
- 很多主题的官方说明会要求安装其他的npm包。使用`npm`安装这些包时，一定要带`--save`选项。这样这些包才会被记录到`package.json`中，这样我们以后push时，GitHub Actions上面的虚拟服务器才会帮我们完整的安装这些依赖。
- 主题本身的配置文件在`themes/<主题名>/_config.yml`中，但这个文件是被主题仓库管理的，不会被hexo仓库管理，无法推送到GitHub。而且主题仓库更新后我们修改的配置也会丢失。所以我们应当在hexo根目录下创建`_config.<主题名>.yml`文件，并把主题中的配置文件拷贝进去。这样配置文件就会被hexo仓库管理。（但是原来主题文件夹里的_config.yml也不要删除）



下面是我安装butterfly的步骤:

1. 把butterfly作为git submodule安装到hexo中

```bash
git submodule add -b master https://github.com/jerryc127/hexo-theme-butterfly.git ./themes/butterfly
```

> 注意：`git submodule add`时，一定要指定主题的文件夹名称，将其命名为主题的名称，此处为`themes/butterfly`。若不这样做，将会变成其GitHub仓库的名称，如`hexo-theme-butterfly`，这样是不行的。

2. 安装渲染器

根据butterfly的官方文档，需要安装渲染器。其他主题可能需要其他渲染器。

```bash
npm install hexo-renderer-pug hexo-renderer-stylus --save
```

> 一定要带`--save`

3. 拷贝**主题配置**文件

把`themes/<主题名>/_config.yml`配置拷到hexo根目录下

```bash
cp themes/butterfly/_config.yml ./_config.butterfly.yml
```

4. 应用主题

修改**hexo配置文件**，`my-hexo`根目录下的`_config.yml`，找到`theme`项并修改：

```yml
theme: butterfly
```

> yml格式的`:`后面一定要有一个空格

5. 查看效果

```bash
hexo clean
hexo g
hexo s --debug
```

![image-20221203191314829](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefinedc9a93fb5a11032c1a3d1c075b0f2e60d.png)

6. 其他修改

根据主题的官方说明，进行其他的自定义配置。所有的修改都在my-hexo根目录下`_config.<主题名>.yml`中修改。 

> `themes/<主题名>/_config.yml`虽然不生效，但是也不能删除。

### 配置hexo

​    hexo本身的配置，需要修改`my-hexo`根目录下的`_config.yml`文件。可以修改站点名称、作者等，具体可以参考官方文档：[配置 | Hexo](https://hexo.io/zh-cn/docs/configuration)。

​    这里提一些与Github Pages还有markdown文章插入相关的配置。

#### 网站url与root

这里一定要把**个人页**和**项目页**分开写清楚：

1. **个人页（user page）**

   仓库名必须是`<用户名>.github.io`，例如我的当前站点：

   `https://jayant-tang.github.io/`

   对应的 Hexo 配置应该是：

```yml
url: https://jayant-tang.github.io
root: /
```

2. **项目页（project page）**

   GitHub 仓库名可以随意，但**最终网址里的路径名默认就是仓库名**。

   例如，如果你的 GitHub 仓库名就叫：

   `blog`

   那么项目页地址通常就是：

   `https://jayant-tang.github.io/blog/`

   对应的 Hexo 配置应该是：

```yml
url: https://jayant-tang.github.io/blog
root: /blog/
```

3. **不要把两种模式配混了**

   - 个人页必须是`root: /`
   - 项目页必须是`root: /<仓库名>/`
   - `url`也要和最终访问地址完全一致
   - 对于项目页，`<仓库名>`不是你自己随便起的一个路径别名，而是你在 GitHub 上创建的那个 Pages 仓库名

我这个博客一开始用的是项目页，后来迁移到了个人页，所以现在实际使用的是第一种配置。

>​    不同环境，访问资源的根目录是不同的。hexo的所有资源文件都应该放在`<hexo文件夹>/source`下，生成网页时，会自动拷贝到网页的根目录下。
>
>​    例如，有一张图片资源的路径是：`<hexo文件夹>/source/foo/bar/image.png`：
>
>- 对于你的markdown文章，可以设置根目录为`<hexo文件夹>/source`
>- 对于网站，则取决于你到底是个人页还是项目页
>
>只要设置好根目录，markdown和网页就可以使用**相同的路径**来访问到**相同的资源**了，如`/foo/bar/image.png`。

#### hexo的部署方式介绍

通常情况下，hexo的部署流程是这样的：

![image-20221203195710806](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined803bfa515d4bb16c7de34b158ce3e96e.png)

1. `hexo g`，生成网页源码，并存储在`/public`文件夹中
2. `hexo d`，把public同步到本地git仓库，然后推送到github仓库。



先安装 git 部署的插件：

```bash
npm install hexo-deployer-git --save
```



然后需要给 Hexo 配置 GitHub 发布仓库的路径和权限，打开**hexo配置文件**：

```yml
deploy:
  type: git
  repo: git@github.com:Jayant-Tang/Jayant-Tang.github.io.git
  branch: master
```

| 选项   | 说明                                              |
| ------ | ------------------------------------------------- |
| type   | 部署类型，可以设置多个要部署的目标                |
| branch | 分支名称                                          |
| repo   | 发布仓库地址，这里有 ssh 和 https 两种方式，推荐 ssh |

>GitHub认证方式：
>
>- https方式，任何人可以随便clone，但是push时通常要走 token 或网页认证；
>- ssh方式，需要ssh key，公钥放在GitHub，私钥自己持有。只要持有私钥就可以用ssh访问GitHub仓库，纯命令行操作，不会弹窗。
>
>我现在本地和 GitHub Actions 都使用 ssh 方式发布。



#### ssh密钥对的生成

​     然后需要生成两对ssh  key，第一对用于**电脑**和**GitHub**之间的认证，第二对用于**GitHub my-hexo仓库** 和**GitHub xxxx.github.io仓库**之间的认证。

​    生成key：

 ```bash
 $ ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/github_ed25519
 # 输入后一直回车
 
 $ ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/hexo_ed25519
 ```

>`-t`：选择加密算法，ed25519比较安全
>
>`-C`:  密钥的备注说明文字。Github要求这里要填上git config时配置的邮箱，同时也必须是GitHub中已经配置的认证邮箱。
>
>`-f`：生成的私钥文件名

执行完毕后，`~/.ssh`下应该有4个新文件，其中带有`.pub`后缀的是公钥，不带的是私钥:

```tex
~/.ssh
|-github_ed25519
|-github_ed25519.pub
|-hexo_ed25519
`-hexo_ed25519.pub
```



#### GitHub ssh key设置

配置你的电脑ssh访问GitHub的权限。GitHub点击右上角头像-Settings，然后点击New SSH key：

然后，把`github_ed25519.pub`的内容粘贴进去，其他都不用填，确认即可：

```bash

```

## 2.5. GitHub Actions自动部署

#### pages仓库添加公钥

#### 编写Actions

在本地`my-hexo`文件夹中，创建一个文件`.github/workflows/main.yml`，填入以下内容：

```bash
name: CI

# 在master分支上收到push时，开始执行流程
on:
  push:
    branches:
      - master

env:
  GIT_USER: Jayant.Tang
  GIT_EMAIL: jayant.tang@nordicsemi.no
  DEPLOY_REPO: Jayant-Tang/Jayant-Tang.github.io
  DEPLOY_BRANCH: master

jobs:
  build:
    name: Build on node ${{ matrix.node_version }} and ${{ matrix.os }}
    runs-on: ubuntu-latest
    strategy:
      matrix:
        os: [ubuntu-latest]
        node_version: [24]

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          submodules: recursive

      - name: Checkout deploy repo
        uses: actions/checkout@v4
        with:
          repository: ${{ env.DEPLOY_REPO }}
          ref: ${{ env.DEPLOY_BRANCH }}
          path: .deploy_git

      - name: Use Node.js ${{ matrix.node_version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node_version }}

      - name: Configuration environment
        env:
          HEXO_DEPLOY_PRI: ${{secrets.HEXO_DEPLOY_PRI}}
        run: |
          sudo timedatectl set-timezone "Asia/Shanghai"
          mkdir -p ~/.ssh/
          echo "$HEXO_DEPLOY_PRI" > ~/.ssh/id_ed25519
          chmod 600 ~/.ssh/id_ed25519
          ssh-keyscan github.com >> ~/.ssh/known_hosts
          git config --global user.name $GIT_USER
          git config --global user.email $GIT_EMAIL

      - name: Install dependencies
        run: |
          npm install

      - name: Deploy hexo
        run: |
          npm run deploy
```

> 内容修改：
>
> 1. 修改环境变量，改成你自己的用户名（`GIT_USER`）、邮箱（`GIT_EMAIL`）、Pages 仓库、Pages 分支
>
> 2. 修改`node_version: [xxxx]`，填上你要使用的 Node.js 版本。我当前环境是 Node.js 24.5.0，所以这里示例写的是`24`
> 	```bash
> 	# 查看本地Node.js版本
> 	node -v
> 	```
>
> 3. 修改时区，如果在中国就不用改了
>
> 内容解释：
>
> 1. 在 hexo 仓库的`master`分支收到 push 时，触发 job
> 2. 在 Ubuntu 环境中 checkout 本仓库的最新内容，并自动获取 submodule
> 3. checkout Pages 发布仓库，并把它放到`.deploy_git`路径中
> 4. 安装指定版本的 Node.js
> 5. 配置环境变量：添加私钥、配置 git 用户和邮箱。注意，这里的私钥名称就是`${{secrets.HEXO_DEPLOY_PRI}}`，与前面配置的`HEXO_DEPLOY_PRI`一致
> 6. 根据`package.json`的内容安装 npm 包
> 7. 执行`npm run deploy`，本质上就是`hexo deploy`
>
> 其他说明：
>
> - 我没有使用别人封装好的自动部署 Hexo 的 Action，而是直接把流程写开。这样 Node.js 版本、ssh、主题、额外脚本都更可控。

# 3. 编写并发布文章

## 3.1. 文章的存储与格式

​    所有要显示的markdown文章都存放在`source/_posts/`目录下，不展示的文章（草稿）都放在`source/_drafts`目录下。

​    新建文章：

```bash
hexo new [layout] <title>
```

> layout: markdown的布局（或者说模板），布局存放在`scaffolds/`目录下，默认有三种布局：
>
> - post.md: 文章模板
> - draft.md：草稿模板
> - page.md：页面模板
>
> title：文章的标题，直接写标题即可，不需要写.md。***但是最好不要有空格等特殊字符！！！***

​	以上命令，会在`source/_posts/`目录下自动创建一个新的`<title>.md`文件，并且把模板内容拷贝过去。由于模板中有`date`参数，这时会自动记录当前的电脑本地时间作为这篇文章的创建时间，记录在Front-Matter中。

## 3.2. Front-Matter

Front-Matter是markdown文件最上方以两行`---`分隔的区域，里面需要填写yml格式的文章配置，例如：

```markdown
---
title: Hello World
date: 2013/7/13 20:46:25
tags: 
 - 嵌入式
 - MCU
---
```

在Typora编辑器中，直接在文章开头输入`---`并回车，就可以创建这个区域：

![image-20221203224247035](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined7bc15927cb8cee959b7937b08c95ed42.png)

​	对于某些主题来说，有些选项是必选的，具体要看对应主题的说明文档。比如，我使用的butterfly主题，就可以配置：

- title：文章标题
- date：文章创建日期
- cover：文章封面图

- tags：文章标签
- categories：文章目录
- ......

>tags和categories的区别：
>
>​    tags是无顺序的，categories是一个递归层级。设计好的tags和categories能让读者很方便的从主页索引到对应的文章。

​    这里，我们也可以体会到前面模板的意义：可以在模板里填好Front Matter，这样每次hexo就会自动帮我们生成了。

## 3.3. Typora在文章中插图的问题

### 方案一：云端图床

图片全部放在云端图床，markdown和Pages都通过URL查看图片。具体的方式，本文不介绍，可以去网上查阅。

### 方案二：图片放在本地

假设我们在markdown中展示一张图片，其路径为`source/imgs/foo/bar/image.png`。

hexo生成网站后，网页会根据 `${root}/imgs/foo/bar/image.png`来找这张图片，即是说，source文件夹放的内容，在网页端就是根目录下的内容。

> 在[2.3 配置hexo](#配置hexo)中，我们已经配置了root为`<仓库名>`了。

​    下面要解决的是，如何在markdown文章和网页中，能用**同样的路径**来访问图片资源。这样本地写文章时和网页端就都可以预览了，而且从markdown转换成html时，路径也是可用的。

​     解决方式就是根目录，由于hexo的根目录前面已经设置过了，我们只需要设置markdown编辑器的根目录就好。设置完毕后，而这就都可以通过`/imgs/foo/bar/image.png`来访问同一张图片了。

**（1）设置markdown文章访问图片时的根目录**

​    我使用的Markdown编辑器是Typora，Typora支持**针对每一篇文章单独设置**根目录。

直接在Front-Matter中填入以下内容即可：

```yml
typora-root-url: ./..
```

> 由于文章的路径是`source/_post`，所以`./..`就是`source`了。

你可以把这一行配置直接写进`scaffolds/post.md`中，这样下次模板就自动帮你填了。

以上设置，直接从菜单栏设置是一样的，Typora会自动帮你写这一行：

![image-20221203221125176](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined9d9d425e0e924da8151e2c19269e660a.png)



**（2）设置Typora粘贴图片时的行为**

直接在Front-Matter中写入：

```yml
typora-copy-images-to: ../../source/imgs/${filename}.assets/
```

这个效果和下图中的偏好设置是一样的，**但是不要去偏好设置里配置，因为那是全局的配置**：

![image-20221203222104681](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefinedfc3be0b10544fff64a8c7f393f0efe76.png)

**（3）检查效果**

​    在Front-Matter中填入以上配置之后，每次粘贴图片到Typora时，就会自动帮你把图片拷贝到`source/imgs/<文章名>.assets/`文件夹中。

## 3.4. Typora中的html标签插图

​    如果你在Typora中右键点击一张图片，然后进行缩放，它就会从Markdown语法的图片:

```markdown
![图片名称](图片路径)
```

​    变成html格式的img标签，比如缩小到50%：

```bash
<img src="图片路径" alt="图片名称" style="zoom: 50%;" />
```

​	hexo在把`.md`文章转换成html的时候，会自动根据我们设置的根目录来替换 markdown 格式图片的路径。

​	这里也要区分**个人页**和**项目页**：

- 如果你是**个人页**，站点根路径就是`/`，那么`![测试图片](/imgs/image.png)`最终还是`/imgs/image.png`
- 如果你是**项目页**，例如`/blog/`，那么`![测试图片](/imgs/image.png)`会被转换成`/blog/imgs/image.png`

​	但是，如果`.md`文章里的图片本来就是 html 格式的标签，**hexo不会对其进行任何转换**。因此：

- 对于**个人页**：`<img src="/imgs/image.png">` 本身就是对的，通常不需要额外处理
- 对于**项目页**：`<img src="/imgs/image.png">` 不会自动变成`/<仓库名>/imgs/image.png`，这时才需要额外处理

我现在的博客已经迁移到**个人页**，所以这一步已经不需要了。  
如果你仍然使用**项目页**，例如仓库名叫`blog`，那可以在 CI 里做一次替换：

``` bash
sed -i 's#<img src="./imgs/#<img src="/blog/imgs/#g' ./source/_posts/*.md
sed -i 's#<img src="/imgs/#<img src="/blog/imgs/#g' ./source/_posts/*.md
sed -i 's#<img src="imgs/#<img src="/blog/imgs/#g' ./source/_posts/*.md
```

​	以上三个命令会把`<img src="/imgs/`、`<img src="./imgs/`和`<img src="imgs/`都替换成`<img src="/blog/imgs/`。

> 默认情况下,html标签中的写法：
>
> - `./imgs`和`imgs`是一样的，在网页端会被当作`/<文章名>/imgs`。
>
> - `/imgs`不会被转换，在网页端还是`/imgs`
>
> 对于项目页，网页上最终能访问到的其实是`/<仓库名>/imgs/...`；对于个人页，则直接是`/imgs/...`。

​    把这些命令加到GitHub Actions中，这样我们本地用typora的时候还是能看到图片，push到远程的时候网页也能看到图片。

​    首先在GitHub Actions的.yml文件中增加一个环境变量，方便后面更改：

<img src="https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined6b73616a62c392fedf467ee0a65edd84.png" alt="image-20221204210311635" style="zoom: 67%;" />

​	然后，在hexo部署的步骤之前，加入sed命令：

```yaml
      - name: Sed img label
        run: |
          sed  -i 's#<img src="./imgs/#<img src="/${{ env.IMG_ROOT }}/imgs/#g'  ./source/_posts/*.md
          sed  -i 's#<img src="/imgs/#<img src="/${{ env.IMG_ROOT }}/imgs/#g'  ./source/_posts/*.md
          sed  -i 's#<img src="imgs/#<img src="/${{ env.IMG_ROOT }}/imgs/#g'  ./source/_posts/*.md
```

​	这样云端部署的时候就会自动帮你替换路径了。

> ​	但是，本地`hexo s --debug`去查看网页的时候，由于没有Actions帮你转换，图片还是看不了的。这个问题倒没什么影响。

# 4. 网站的其他设置

## 基本概念

​    博客除了**文章**外，还有**页面**。页面的模板就是`scaffolds/post.md`。可以有时间线、标签、目录等页面，页面也存放在`source`目录下。

​    在`_config.yml`中，配置了这些页面的目录：

​    基本上所有的配置你都可以在[配置 | Hexo](https://hexo.io/zh-cn/docs/configuration) 和你所用的主题的官方文档中找到。

## 文章的永久链接（Permalinks）

​	hexo默认的文章URL格式是`/年/月/日/文章标题`，如果你的文章标题有中文，这个URL就会巨长无比：

![image-20221204210811289](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined3062b0a3d9c71266d12b0365c65d15a9.png)

​	我的方案是改成用文章标题+日期的hash值去命名url，这样更简洁。

​	打开**站点配置**`_config.yml`，修改：

```yaml
permalink: :year/:month/:hash/
```

​	这样，文章的URL就会变成：

![image-20221204211202297](https://jayant-blog-imgs.oss-cn-hangzhou.aliyuncs.com/undefined5155c884c3388f5e5c50b5c84f677569.png)

​	这样URL就比较短了。



## 文章的更新日期

​	文章具有**创建日期（date）**和**更新日期（updated）**。默认情况下，更新日期使用的是操作系统中记录的文件最后修改日期。

​	但是我们利用Github Actions部署时，由于每次部署都会重新拉取仓库，文件的修改日期会变成拉取的时间，导致所有文章的更新日期都会变成部署时间。



## 其他注意事项

- 所有文章标题、文件名、目录名都不要有空格等特殊字符！！！（可以有减号`-`）
- 本博客使用了Gitalk作为评论系统，把页面仓库的Github Issue变成一个评论区。会有一些API key之类的东西，明文配置在hexo仓库中，所以我把hexo仓库的权限设为私有了
- Front-Matter是yml格式，冒号`:`后面一定要加一个空格，不然识别不了。
